# 인테리어 전체 분야 견적 요청 시스템 설계안

> 기준 문서: `견적_양식_현황.md` (현재 = 쇼파 전용, 5단계 Wizard, 단일 플랫 테이블)
> 목표: 쇼파 전용 → 인테리어 전체 분야로 확장. 복수 분야 선택, 분야별 동적 폼/렌더링, 임시저장, 분야별 업체 매칭까지 고려한 운영 가능한 설계.

---

## A. 현재 구조의 문제점 요약

| # | 문제 | 영향 |
|---|------|------|
| 1 | Step 2~4가 쇼파 전용 필드에 종속 (`sofa_type`, `seat_count`, `backrest_height` 등) | 다른 분야를 받을 수 없음. 분야 추가 = 폼 전면 수정 |
| 2 | `quote_requests` 단일 플랫 테이블 | 분야 늘수록 컬럼 폭증 + NULL 다수. 쿼리·인덱스·통계 모두 악화 |
| 3 | `category` 같은 분야 구분 컬럼 없음 | "모든 요청 = 쇼파" 암묵적 가정. 분야별 필터/매칭/통계 불가 |
| 4 | 고객/공장/관리자 상세 화면 3곳에 쇼파 필드 하드코딩 | 분야 추가 시 3곳을 매번 수정. 동적 렌더링 불가 |
| 5 | 서버 액션이 `...data` 객체 전체를 스프레드로 INSERT | 폼 키 = DB 컬럼명이 1:1로 강결합. 구조 바꾸기 어려움 |
| 6 | 복수 분야(예: 바닥+도장+조명) 선택 불가 | 실제 인테리어 견적은 다분야가 기본인데 1요청=1품목 구조 |
| 7 | 임시저장이 localStorage 단독 (`sofit_quote_draft`) | 기기 변경·캐시 삭제 시 소실. 서버에 작성중 상태 없음 |
| 8 | 폼 라벨 mm vs DB 주석 cm 단위 불일치 | 데이터 신뢰성 저하 (확장 전에 정리 필요) |

**근본 원인 한 줄 요약**: "분야별로 다른 데이터"를 *컬럼*과 *하드코딩 JSX*로 표현하고 있어서, 분야가 늘 때마다 DB·폼·뷰·타입·서버액션 5곳을 동시에 고쳐야 한다. → 이걸 **스키마(데이터) 기반 + JSONB 저장 + 동적 렌더링**으로 바꾸는 것이 설계의 전부다.

## B. 인테리어 전체 분야 카테고리 체계

분야를 단순 나열하지 않고 **공정 흐름(철거→골조→설비→마감→가구→설치→마무리)** 순서의 대분류로 묶었다. 이 순서는 그대로 좌측 사이드바 정렬 순서로 쓸 수 있다.

| 대분류(group) | 분야(category) | category key |
|---|---|---|
| **1. 철거·기초** | 철거 | `demolition` |
| | 폐기물 처리 | `waste` |
| | 설비(급배수·배관) | `plumbing` |
| | 방수 | `waterproofing` |
| **2. 골조·목공** | 목공(가벽·천장·틀) | `carpentry` |
| | 금속(난간·프레임·철물) | `metal` |
| | 유리(파티션·도어·거울) | `glass` |
| | 창호(샷시·발코니창) | `window` |
| | 중문 | `interior_door` |
| **3. 마감** | 바닥(마루·장판·폴리싱) | `flooring` |
| | 타일 | `tile` |
| | 도배 | `wallpaper` |
| | 도장(페인트) | `painting` |
| | 필름·시트 | `film` |
| **4. 설비·전기** | 전기·배선 | `electrical` |
| | 조명 | `lighting` |
| | 냉난방(에어컨·난방) | `hvac` |
| **5. 공간 가구** | 주방 | `kitchen` |
| | 욕실 | `bathroom` |
| | 붙박이·수납 | `builtin` |
| **6. 가구·소품** | 쇼파·가구 | `sofa` |
| | 커튼·블라인드 | `curtain` |
| | 가전 | `appliance` |
| **7. 사인·외부** | 간판 | `signage` |
| | 사인물(실내사인·시트) | `signboard` |
| | 조경·플랜테리어 | `landscaping` |
| **8. 마무리** | 청소(입주·준공) | `cleaning` |

> 총 27개 분야. 시작은 이 중 빈도 높은 8~10개만 스키마를 완성해 오픈하고, 나머지는 "상담 신청"만 받는 **경량 스키마**로 두었다가 점진 확장하면 된다(J. 구현 우선순위 참고).

## C. 공통 견적 요청 필드 (모든 분야 공유)

좌측에서 어떤 분야를 고르든 항상 받는 정보. 현재 Step 1·5를 재정비한 것이다.

### C-1. 현장 기본 정보
| 키 | 라벨 | 타입 | 필수 | 모름 옵션 | 비고 |
|---|---|---|---|---|---|
| `site_name` | 현장명 | text | ✅ | — | |
| `address` | 현장 주소 | address | ✅ | — | 우편번호+상세 분리 권장 |
| `contact_name` | 담당자명 | text | ✅ | — | |
| `contact_phone` | 연락처 | tel | ✅ | — | |
| `building_type` | 건물 유형 | select | ✅ | — | 주거(아파트/빌라/주택) / 상가 / 사무실 / 숙박 / 기타 |

| `has_elevator` | 엘리베이터 | boolean | ❌ | 자재 반입 동선 판단 |

| `available_time` | 방문/공사 가능 시간 | text | ❌ | |

### C-3. 일정·예산·요청 (Step 5 + 신규)
| 키 | 라벨 | 타입 | 필수 | 모름 옵션 | 비고 |
|---|---|---|---|---|---|

| `needs_measurement` | 실측 필요 | boolean | ❌ | | 체크 시 분야별 치수는 "실측 요청" 처리 |

## D. 분야별 견적 요청 필드

> 형식: `필수` = 견적 산정에 반드시 필요 / `선택` = 없어도 진행 가능 / `모름옵션` = 전문 용어를 몰라도 넘어갈 수 있게 "모름 / 상담 필요 / 현장 확인 필요" 선택지 제공.
> 모든 분야 공통으로 `note`(분야별 메모, textarea, 선택)와 `reference_images`(우측 첨부, 선택)를 갖는다.
> 아래 필드는 **그대로 K의 스키마 레지스트리(`FIELD_SCHEMAS`)와 1:1 매핑**된다.

### 쇼파·가구 `sofa` (기존 자산 이식)
- **필수**: 형태(직선/ㄱ자/ㄷ자/원형/기타), 수량
- **선택**: 좌석 수, 등받이 높이(하이/미디/로우), 쿠션(스프링/폼/혼합), 프레임(목재/철재/혼합), 원단(패브릭/가죽/인조/벨벳), 색상, 방염·방수 여부, 전체 L/W/H·좌면 높이/깊이(mm)
- **모름옵션**: 원단·치수 → "모름/추천 요청", "실측 요청"

### 철거 `demolition`
- **필수**: 철거 범위(전체/부분/특정 구간), 철거 대상(벽체·바닥·천장·주방·욕실·설비 다중선택)
- **선택**: 철거 면적, 잔재 처리 포함 여부
- **모름옵션**: 내력벽 포함 여부 → **"모름/현장 확인 필요"** (구조 안전 직결), 석면 의심 → "모름/확인 필요"

### 폐기물 처리 `waste`
- **필수**: 폐기물 종류(건축/생활/혼합), 예상 배출량(소량/1톤/2.5톤/5톤↑)
- **선택**: 반출 동선(엘베/계단/사다리차 필요), 분리배출 여부
- **모름옵션**: 배출량 → "모름"

### 설비 `plumbing`
- **필수**: 작업 종류(급수/배수/난방배관/위치 이동 다중선택)
- **선택**: 보일러 관련 작업, 배관 위치
- **모름옵션**: 누수 여부·배관 노후 → "모름/현장 확인 필요"

### 방수 `waterproofing`
- **필수**: 시공 부위(욕실/베란다/옥상/주방/외벽 다중선택)
- **선택**: 면적
- **모름옵션**: 현재 누수 현황 → "모름/현장 확인 필요"

### 목공 `carpentry`
- **필수**: 작업 항목(가벽/천장/문틀·몰딩/가구틀/파티션/단상 다중선택), 작업 공간
- **선택**: 천장 형태(평/우물/간접), 면적·길이, 마감 포함 여부
- **모름옵션**: 자재 등급(일반/친환경 E0) → "모름/추천 요청"

### 금속 `metal`
- **필수**: 작업 항목(난간/계단/프레임/철문/구조 보강 다중선택)
- **선택**: 자재(스테인리스/철/알루미늄), 도장 포함, 사이즈
- **모름옵션**: 자재 → "모름/추천 요청"

### 유리 `glass`
- **필수**: 작업 항목(파티션/도어/샤워부스/거울/쇼윈도 다중선택)
- **선택**: 유리 종류·두께, 강화/접합 여부, 사이즈
- **모름옵션**: 종류·사이즈 → "모름", "실측 요청"

### 창호 `window`
- **필수**: 창호 종류(이중창/시스템창/발코니창/현관문 다중선택), 개소(수량)
- **선택**: 규격(가로×세로), 단열 등급, 색상, 방충망 포함
- **모름옵션**: 규격 → "실측 요청", 단열 등급 → "모름/추천"

### 중문 `interior_door`
- **필수**: 종류(3연동/슬라이딩/스윙/폴딩), 개소
- **선택**: 유리 종류(투명/모루/망입), 프레임 색상, 손잡이 방향
- **모름옵션**: 사이즈 → "실측 요청"

### 바닥 `flooring`
- **필수**: 바닥재 종류(강마루/강화/원목/장판/타일/폴리싱/에폭시/데코타일), 시공 면적
- **선택**: 기존 바닥 철거 여부, 패턴/방향, 걸레받이 포함, 평탄 작업 필요
- **모름옵션**: 종류 → "모름/추천 요청", 난방 종류(온수/전기/없음) → "모름"

### 타일 `tile`
- **필수**: 시공 부위(바닥/벽/욕실/주방/현관 다중선택), 면적
- **선택**: 규격(300각/600각/대형), 종류(자기질/도기질/포세린), 줄눈 색상, 패턴(일반/헤링본)
- **모름옵션**: 규격·종류 → "모름/추천", 기존 타일 철거 vs 덧방 → "모름/현장 확인"

### 도배 `wallpaper`
- **필수**: 종류(실크/합지/방염/뮤럴), 시공 면적(또는 "벽면 기준")
- **선택**: 기존 벽지 제거 여부, 천장 포함, 컬러/패턴 희망
- **모름옵션**: 종류 → "모름/추천", 곰팡이·결로 부위 → "모름/현장 확인"

### 도장 `painting`
- **필수**: 도장 부위(벽/천장/문·몰딩/외부/철물 다중선택), 면적
- **선택**: 페인트 종류(수성/유성/친환경/방수/특수), 색상(지정/추천 요청)
- **모름옵션**: 종류 → "모름/추천", 퍼티·바탕 작업 필요 → "모름/현장 확인"

### 필름·시트 `film`
- **필수**: 시공 부위(문·문틀/가구/벽/창틀 다중선택), 면적 또는 수량
- **선택**: 컬러/패턴, 기존 표면 상태
- **모름옵션**: 필름 등급/브랜드 → "모름/추천 요청"

### 전기 `electrical`
- **필수**: 작업 종류(콘센트·스위치 증설/배선/분전반/통신·네트워크 다중선택)
- **선택**: 증설 위치·수량, 노후·누전 점검 필요
- **모름옵션**: 용량 증설 필요 여부 → "모름/현장 확인 필요"

### 조명 `lighting`
- **필수**: 작업 범위(전체 교체/부분/신규 설치), 적용 공간
- **선택**: 조명 종류(매입등/직부/레일/펜던트/간접/라인 다중선택), 수량, 디밍·스마트 여부
- **모름옵션**: 종류·수량·색온도(전구색/주백색/주광색) → "모름/추천 요청"

### 냉난방 `hvac`
- **필수**: 작업 종류(시스템에어컨/벽걸이/스탠드/바닥난방 다중선택), 공간 수 또는 평수
- **선택**: 실외기 위치, 배관 신설 여부
- **모름옵션**: 용량(평형) → "모름/추천 요청"

### 주방 `kitchen`
- **필수**: 작업 범위(전체 교체/부분/싱크대만), 주방 형태(일자/ㄱ자/ㄷ자/아일랜드)
- **선택**: 상부장·하부장 길이, 상판 재질(인조대리석/세라믹/스테인리스), 후드·쿡탑 포함, 수납 요구
- **모름옵션**: 길이 → "실측 요청", 형태·상판 → "모름/추천"

- **필수**: 작업 범위(전체/부분), 욕실 개수

- **모름옵션**: 방수 포함 여부 → "모름/상담 필요", 사이즈 → "실측 요청"

### 붙박이·수납 `builtin`
- **필수**: 가구 종류(붙박이장/신발장/팬트리/책장/TV장 다중선택), 설치 위치
- **선택**: 도어 형태(여닫이/슬라이딩), 자재·색상, 내부 구성(선반/서랍/행거)
- **모름옵션**: 사이즈 → "실측 요청"

### 커튼·블라인드 `curtain`
- **필수**: 종류(커튼/블라인드/롤스크린/허니콤/버티컬 다중선택), 창 개수
- **선택**: 전동 여부, 암막 여부, 원단·색상
- **모름옵션**: 창 사이즈 → "실측 요청"

### 가전 `appliance`
- **필수**: 가전 종류(다중선택), 설치 방식(일반 설치/빌트인)
- **선택**: 모델·스펙, 전기·배관 추가 작업 필요
- **모름옵션**: 모델 → "모름/추천 요청"

- **필수**: 간판 종류(전면/돌출/입간판/LED 채널 다중선택), 설치 위치
- **선택**: 사이즈(가로×세로), 조명 방식(채널/LED/무조명), 브랜드 가이드 유무
- **모름옵션**: 디자인 보유 여부(있음/제작 필요), 전기 인입 → "모름/현장 확인", 사이즈 → "실측 요청"

### 사인물 `signboard`
- **필수**: 종류(실내 사인/유도 사인/네임택/시트지/배너 다중선택)
- **선택**: 수량, 사이즈
- **모름옵션**: 디자인 제작 필요 여부 → "보유 / 제작 요청"

## E. 추천 UI 구조

데스크톱은 3분할, 모바일은 탭 전환으로 같은 정보 구조를 재배치한다.

```
┌──────────────┬──────────────────────────────┬──────────────────┐
│  좌: 분야 선택 │      중앙: 현재 작성 폼         │  우: 첨부 + 요약    │
│  + 진행률      │  (선택한 분야 순서대로 섹션)     │                  │
├──────────────┼──────────────────────────────┼──────────────────┤
│ □ 공통 정보 ✅ │  [공통 정보]                  │ 📎 공통 첨부        │
│ ─────────    │   현장명 / 주소 / 면적 ...      │  (현장 사진·도면)   │
│ ☑ 목공  60%  │                              │                  │
│ ☑ 도장  ✅    │  [목공]  ← 현재 섹션           │ 📎 목공 첨부        │
│ ☑ 조명  20%  │   작업 항목 / 공간 ...          │ 📎 도장 첨부        │
│ ☑ 간판  0%   │   [다음 분야 ▶]               │ 📎 조명 첨부        │
│ □ 청소       │                              │ ───────          │
│ ─────────    │                              │ 📋 요청 요약        │
│ + 분야 추가   │                              │  분야 4개 / 첨부 7   │
│              │                              │  [임시저장됨 ·방금]  │
└──────────────┴──────────────────────────────┴──────────────────┘
```

### 좌측 사이드바
- **공통 정보** 항목이 맨 위 고정(항상 작성). 그 아래 **전체 분야 목록**을 대분류 그룹으로 펼침/접기.
- 분야는 **체크박스 복수 선택**. 선택하면 중앙·우측에 해당 섹션이 즉시 추가/제거.
- 각 분야 옆에 **작성 진행률(%) 또는 ✅** 표시 → 한눈에 "뭘 더 채워야 하는지" 파악.
- 미선택 분야는 흐리게 + "추가" 액션.

### 중앙 작성 영역
- 선택한 분야를 **세로 스크롤 섹션**으로 쌓는다(아코디언). Wizard식 5스텝 강제 이동 대신, 현재 분야 섹션에 집중하되 위아래로 자유 이동.
- 각 필드는 스키마 기반 자동 렌더링(D의 type → 입력 위젯 매핑).
- "모름/상담/실측" 옵션은 각 입력 옆 작은 토글 또는 select의 마지막 항목으로 제공.
- 섹션 하단에 "이 분야 작성 완료 → 다음 분야" 버튼.

### 우측 참고 이미지/요약
- **공통 첨부**(전체 현장 사진·도면·기존 견적서) + **분야별 첨부**(간판 디자인, 바닥 레퍼런스 등)를 분리 노출.
- 드래그&드롭 업로드, 썸네일, 분야 태그 표시.
- 하단에 **실시간 요청 요약**(선택 분야 수, 필수 미입력 개수, 첨부 수) + **임시저장 상태 배지**("방금 저장됨").

### 단계별 작성 Flow (Wizard → 자유 섹션 하이브리드)
1. **분야 선택**(좌측) → 2. **공통 정보**(중앙 맨 위) → 3. **분야별 작성**(분야 순서대로) → 4. **검토·요약**(우측 요약 → 제출 전 미입력 필수 하이라이트) → 5. **제출**.
- 진행 강제는 하지 않되, 제출 시 필수 미입력 분야로 자동 스크롤·하이라이트.

### 임시저장 UX
- 입력 변경 후 **1.5초 디바운스 자동 저장**(서버) + 즉시 localStorage 백업.
- 우측 상단에 "저장 중… / 방금 저장됨 / 저장 실패(재시도)" 상태 표시.
- 재진입 시 "작성 중이던 견적이 있습니다 — 이어서 작성 / 새로 시작" 모달.

### 모바일
- 하단 탭 3개: **[분야] · [작성] · [첨부/요약]**. 좌·우 사이드바 내용을 탭으로 전환.
- 분야 선택은 바텀시트, 작성은 단일 컬럼 풀스크린, 첨부는 별도 탭.
- 진행률은 상단 고정 프로그레스 바.

## F. 추천 User Flow

```
[신규 견적 작성 시작]
        │  서버: quote_requests INSERT (status='draft') → request_id 발급
        ▼
[좌측에서 분야 복수 선택]  목공 + 도장 + 조명 + 간판 + 청소
        │  선택/해제 시 quote_request_items upsert/delete (status='draft')
        ▼
[공통 정보 작성]  ──자동저장(debounce)──▶ quote_requests UPDATE
        ▼
[분야별 작성]  목공→도장→조명→…  ──자동저장──▶ items[*].details(JSONB) UPDATE
        ▼
[이미지 첨부]  공통 첨부 / 분야별 첨부  ──업로드──▶ Storage + attachments INSERT(item_id 유무로 스코프)
        ▼
[임시저장]  새로고침·이탈해도 draft 유지 (서버 row + localStorage)
        ▼
[검토]  우측 요약에서 필수 미입력 확인 → 제출 버튼
        │  서버 검증(스키마 기반): 분야별 required 충족?
        ▼
[제출]  quote_requests.status='submitted', submitted_at 기록
        │  → 분야별로 matches 생성(분야별 업체 풀과 매칭)
        │  → 관리자 notifications, chat_room 생성
        ▼
[제출 후]
  · 고객: 요청 상세(읽기) + 진행 상태
  · 관리자: 분야별 탭으로 정리된 상세 + 분야별 업체 배정
  · 업체(공장): 자신이 속한 분야 item만 노출 → 분야별 견적서 작성
```

핵심: **요청 시작 시점에 draft row를 먼저 만든다.** 그래야 자동저장·첨부·복수 분야가 전부 동일 `request_id`에 묶이고, 제출은 단지 `status` 전환이 된다.

## G. 추천 DB 구조

### G-1. 세 방식 비교 (요구사항 기준 평가)

| 기준 | (A) 컬럼 계속 추가 | (B) 공통 컬럼 + 단일 JSONB | (C) 부모 + items 자식 테이블 (+ JSONB) |
|---|---|---|---|
| 분야 확장(스키마 변경 없이) | ❌ 매번 ALTER | ✅ | ✅ |
| **복수 분야 선택(1요청 N분야)** | ❌ | △ JSONB 배열로 가능하나 분야 단위 처리 불편 | ✅ **자식 행 N개** |
| **분야별 업체 매칭** | ❌ | ❌ 매칭이 분야 단위인데 부모에 뭉쳐 있음 | ✅ item_id 단위 매칭 |
| 분야별 견적서 비교 | ❌ | △ | ✅ factory_quotes → item_id |
| 통계/검색(분야별 집계) | △ NULL 지옥 | △ JSONB 풀어야 함 | ✅ category 컬럼 인덱스 + GIN |
| 관리자 동적 렌더링 | ❌ 하드코딩 | ✅ | ✅ |
| 구현 복잡도 | 낮음 | 낮음 | 중간 |

→ **결론: (C)를 뼈대로, 분야별 상세는 (B)의 JSONB로.** 즉 **하이브리드**.
- (C)가 주는 것: 복수 분야 1:N, 분야별 매칭·견적 비교, category 인덱스 기반 통계.
- (B)가 주는 것: 분야 추가 시 스키마 변경 0 (JSONB에 담음).
- 트레이드오프(쿼리·표시 복잡)는 **스키마 레지스트리 + 동적 렌더러**로 흡수.

### G-2. 테이블 설계

```sql
-- 1) 견적 요청 (부모): 공통 정보 + 상태
CREATE TABLE quote_requests (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  customer_id     uuid NOT NULL REFERENCES customers(id),
  status          text NOT NULL DEFAULT 'draft'
                    CHECK (status IN ('draft','submitted','matched','closed','cancelled')),
  -- 공통 현장 정보 (C-1, C-2)
  site_name       text,
  address         jsonb,            -- { zipcode, road, detail, lat, lng }
  contact_name    text,
  contact_phone   text,
  building_type   text,
  space_status    text,
  area_m2         numeric,          -- 내부 ㎡ 통일 (UI는 평 토글)
  business_type   text,
  floor           int,
  total_floors    int,
  has_elevator    boolean,
  has_parking     boolean,
  is_occupied     text,
  available_time  text,
  -- 일정·예산 (C-3)
  desired_start_date date,
  desired_end_date   date,
  needs_measurement  boolean DEFAULT false,
  budget_range       text,
  special_requests   text,
  -- 상태/이력
  submitted_at    timestamptz,
  admin_note      text,
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now()
);

-- 2) 분야별 항목 (자식): 한 요청에 여러 분야 = 여러 행
CREATE TABLE quote_request_items (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  request_id    uuid NOT NULL REFERENCES quote_requests(id) ON DELETE CASCADE,
  category      text NOT NULL,        -- 'carpentry','painting','signage' ... (B의 category key)
  schema_version int NOT NULL DEFAULT 1, -- 스키마 버전 (필드 바뀌어도 과거 데이터 해석 가능)
  details       jsonb NOT NULL DEFAULT '{}',  -- 분야별 입력값 (D의 필드)
  status        text NOT NULL DEFAULT 'draft',
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now(),
  UNIQUE (request_id, category)        -- 같은 요청에 같은 분야 중복 방지
);
CREATE INDEX idx_qri_request   ON quote_request_items (request_id);
CREATE INDEX idx_qri_category  ON quote_request_items (category);
CREATE INDEX idx_qri_details   ON quote_request_items USING gin (details);  -- JSONB 검색/통계

-- 3) 첨부파일: 공통(item_id NULL) 또는 분야별(item_id 지정)
CREATE TABLE quote_request_attachments (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  request_id   uuid NOT NULL REFERENCES quote_requests(id) ON DELETE CASCADE,
  item_id      uuid REFERENCES quote_request_items(id) ON DELETE CASCADE, -- NULL = 공통 첨부
  kind         text NOT NULL,        -- 'site_photo','style_ref','drawing','sketch','prev_quote','etc'
  storage_path text NOT NULL,        -- Supabase Storage 경로
  file_name    text,
  mime_type    text,
  size_bytes   bigint,
  created_at   timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX idx_qra_request ON quote_request_attachments (request_id);
CREATE INDEX idx_qra_item    ON quote_request_attachments (item_id);

-- 4) 수정 이력 (선택, 운영용)
CREATE TABLE quote_request_revisions (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  request_id  uuid NOT NULL REFERENCES quote_requests(id) ON DELETE CASCADE,
  snapshot    jsonb NOT NULL,        -- 제출/주요 변경 시점의 전체 스냅샷
  reason      text,                  -- 'submitted','customer_edit','admin_edit'
  created_by  uuid,
  created_at  timestamptz NOT NULL DEFAULT now()
);

-- 매칭/견적서는 분야(item) 단위로 연결
ALTER TABLE matches        ADD COLUMN item_id uuid REFERENCES quote_request_items(id);
ALTER TABLE factory_quotes ADD COLUMN item_id uuid REFERENCES quote_request_items(id);
```

### G-3. JSONB `details` 예시 (간판 item)

```json
{
  "signage_types": ["front", "led_channel"],
  "install_location": "건물 1층 전면 상단",
  "size": { "width_mm": 4000, "height_mm": 800, "unknown": false },
  "lighting_method": "led_channel",
  "has_design": "need_design",          // owned | need_design
  "electricity_inlet": "site_check",    // yes | no | site_check (현장 확인)
  "note": "기존 간판 철거 후 동일 위치 설치 희망"
}
```

> "모름/상담/현장확인"은 값 자체로 표현(`"site_check"`, `"unknown": true`, `"need_design"`). 그래야 업체·관리자가 "고객이 모른다"는 사실까지 데이터로 받는다.

### G-4. draft / submitted 상태 관리
- `quote_requests.status`: `draft` → `submitted` → `matched` → `closed`. **임시저장 = draft row가 그대로 존재**. 제출은 status 전환 + `submitted_at` 기록 + 제출 시점 스냅샷을 `revisions`에 저장.
- 자동저장은 draft 상태에서 `UPDATE`(upsert)만 수행. 제출 후 수정은 새 revision으로 누적.
- 한 고객의 진행 중 draft는 1건으로 제한(재진입 시 이어쓰기)하거나, 복수 허용 시 "임시저장 목록" 제공.

## H. 프론트엔드 컴포넌트 구조 (React / Next.js App Router)

## I. 백엔드 / 서버 액션 구조

```
src/app/actions/quote-request.ts
├─ ensureDraft()                 # 진입 시 draft 없으면 생성, request_id 반환
├─ saveCommonInfo(requestId, patch)   # 공통 정보 부분 저장 (자동저장)
├─ upsertCategoryItem(requestId, category, detailsPatch)  # 분야 추가/상세 저장
├─ removeCategoryItem(requestId, category)                # 분야 해제
├─ uploadAttachment(requestId, itemId|null, file, kind)   # Storage 업로드 + 메타 INSERT
├─ removeAttachment(attachmentId)
├─ submitQuoteRequest(requestId)      # 검증 → status='submitted' → 매칭/알림 트리거
└─ getQuoteRequest(requestId)         # 부모+items+attachments 조인 조회
```

설계 원칙:
- **`...data` 스프레드 INSERT 폐기.** 공통 필드는 화이트리스트로 명시 매핑, 분야 상세는 `details` JSONB에 담는다. (현황 문서의 강결합 문제 해소)
- **검증은 스키마 단일 소스로.** `submitQuoteRequest`가 `FIELD_SCHEMAS[item.category]`의 `required`를 순회하며 `details`에 값이 있는지 검사. 프론트·백엔드 동일 스키마 사용 → 검증 로직 중복 제거.
- **자동저장은 가볍게.** `saveCommonInfo` / `upsertCategoryItem`은 부분 패치 upsert만. 실패해도 localStorage가 보존.
- **파일 업로드는 별도 액션.** 폼 저장과 분리(대용량·실패 격리). 업로드 성공 후 메타만 DB에 기록, `item_id` 유무로 공통/분야 스코프 결정.
- **제출은 트랜잭션.** status 전환 + revision 스냅샷 + 분야별 matches 생성 + 알림을 한 트랜잭션으로.

**임시저장: 로컬 vs 서버 — 권장 = 둘 다(서버 우선).**
| 방식 | 장점 | 단점 |
|---|---|---|
| localStorage 단독(현재) | 즉시·오프라인·무료 | 기기 변경/캐시 삭제 시 소실, 관리자가 작성중 현황 못 봄 |
| 서버 단독 | 어디서나 이어쓰기, 운영 가시성 | 네트워크 의존, 매 타이핑 저장 부담 |
| **하이브리드(권장)** | 즉시성(local) + 영속성(server) | 동기화 로직 필요 |

→ **서버를 진실 공급원으로, localStorage는 즉시 백업/오프라인 캐시.** 입력 시 local 즉시 기록 + 1.5초 debounce로 server upsert. 재진입 시 server draft 로드 후, local에 "서버보다 최신" 변경이 있으면 병합.

## J. 구현 우선순위

### 1차 MVP (먼저 동작하게)
1. DB 마이그레이션: `quote_requests`(공통 재정비) + `quote_request_items` + `attachments`.
2. 스키마 레지스트리 `FIELD_SCHEMAS` — **빈도 높은 8개 분야만** 완성: 목공·바닥·도배·도장·조명·주방·욕실·쇼파(기존 이식).
3. `QuoteRequestStudio` 3분할 폼: 분야 복수 선택 + 공통 + 분야별 동적 렌더 + 제출.
4. 서버 액션: `ensureDraft`, `saveCommonInfo`, `upsertCategoryItem`, `submit`.
5. 상세 3화면을 `CategoryItemView` 기반 동적 렌더로 교체(하드코딩 제거).
6. 기존 쇼파 데이터 마이그레이션 스크립트(플랫 컬럼 → `category='sofa'` item.details).

### 2차 고도화
1. 나머지 19개 분야 스키마 추가(스키마만 추가하면 폼·뷰 자동 반영).
2. 하이브리드 자동저장(local+server 병합) + 재진입 모달.
3. 첨부 분야별 스코프 + 갤러리, 도면/이미지 미리보기.
4. 제출 검증 강화, "모름/상담/실측" 항목을 관리자 화면에서 강조 표시.
5. 분야별 업체 매칭(matches.item_id) + 분야별 견적서(factory_quotes.item_id) 비교.

### 3차 운영/자동화
1. 스키마를 코드 → DB(`category_field_schemas`) 이전 → 비개발자(운영팀)가 필드 편집.
2. JSONB GIN 인덱스 기반 통계·검색(분야별 요청 추이, 평균 예산, 미입력률).
3. 견적 비교 대시보드, 분야별 자동 업체 추천(과거 매칭·평점 기반).
4. 수정 이력(`revisions`) 기반 변경 추적·복원.

```tsx
// components/DynamicField.tsx
function DynamicField({ def, value, onChange }: { def: FieldDef; value: any; onChange: (v:any)=>void }) {
  const isUnknown = value?.__unknown === true;
  return (
    <div className="field">
      <label>{def.label}{def.required && <span className="req">*</span>}</label>
      {def.help && <p className="help">{def.help}</p>}

      {!isUnknown && (() => {
        switch (def.type) {
          case 'text':     return <input value={value ?? ''} onChange={e=>onChange(e.target.value)} />;
          case 'number':   return <input type="number" value={value ?? ''} onChange={e=>onChange(+e.target.value)} />;
          case 'textarea': return <textarea value={value ?? ''} onChange={e=>onChange(e.target.value)} />;
          case 'boolean':  return <Toggle checked={!!value} onChange={onChange} />;
          case 'date':     return <input type="date" value={value ?? ''} onChange={e=>onChange(e.target.value)} />;
          case 'select':   return <Select options={def.options!} value={value} onChange={onChange} />;
          case 'multiselect': return <MultiSelect options={def.options!} value={value ?? []} onChange={onChange} />;
          case 'dimension':   return <DimensionInput unit={def.unit} value={value} onChange={onChange} />;
        }
      })()}

      {/* "모름 / 상담 / 현장확인 / 실측" 옵션 */}
      {def.unknown && (
        <UnknownToggle
          mode={def.unknown}              // consult | site_check | measure | unknown
          active={isUnknown}
          onToggle={(on)=> onChange(on ? { __unknown: true, reason: def.unknown } : null)}
        />
      )}
    </div>
  );
}
```

### K-3. 자동저장 훅 (server upsert + local 백업)

```typescript
// hooks/useAutosave.ts
function useAutosave(requestId: string) {
  const save = useDebouncedCallback(async (kind: 'common'|'item', payload: any) => {
    localStorage.setItem(`quote_draft_${requestId}`, JSON.stringify(getFullState())); // 즉시 백업
    setStatus('saving');
    try {
      if (kind === 'common') await saveCommonInfo(requestId, payload);
      else await upsertCategoryItem(requestId, payload.category, payload.details);
      setStatus('saved');
    } catch { setStatus('error'); /* local 백업은 유지됨 */ }
  }, 1500);
  return save;
}
```

### K-4. 제출 서버 액션 (스키마 기반 검증)

```typescript
// actions/quote-request.ts
export async function submitQuoteRequest(requestId: string) {
  const { request, items, attachments } = await loadRequest(requestId);

  // 1) 공통 필수 검증
  const commonErrors = validateAgainst(COMMON_SCHEMA, request);

  // 2) 분야별 필수 검증 (스키마 단일 소스)
  const itemErrors = items.flatMap(item => {
    const schema = FIELD_SCHEMAS[item.category];
    return schema.fields
      .filter(f => f.required)
      .filter(f => {
        const v = item.details[f.key];
        const filled = v !== undefined && v !== null && v !== '' &&
                       !(Array.isArray(v) && v.length === 0);
        const unknownOk = v?.__unknown === true;   // "모름" 선택도 통과로 인정
        return !filled && !unknownOk;
      })
      .map(f => ({ category: item.category, key: f.key, label: f.label }));
  });

  if (commonErrors.length || itemErrors.length) {
    return { ok: false, commonErrors, itemErrors }; // 프론트가 해당 필드로 스크롤·하이라이트
  }

  // 3) 트랜잭션: 상태 전환 + 스냅샷 + 분야별 매칭 + 알림
  await db.transaction(async (tx) => {
    await tx.update(quote_requests).set({ status: 'submitted', submitted_at: now() }).where(eq(quote_requests.id, requestId));
    await tx.insert(quote_request_revisions).values({ request_id: requestId, snapshot: { request, items, attachments }, reason: 'submitted' });
    for (const item of items) {
      const factories = await findFactoriesByCategory(tx, item.category); // 분야별 업체 풀
      await createMatches(tx, requestId, item.id, factories);             // item 단위 매칭
    }
    await notifyAdmins(tx, requestId);
  });

  return { ok: true };
}
```

## 한 줄 정리

> **데이터 모델(부모 + 분야 items + JSONB) · 스키마 레지스트리(FIELD_SCHEMAS) · 동적 렌더러(DynamicField / CategoryItemView)** 이 세 가지가 축이다.
> 분야가 27개로 늘든 100개로 늘든, **새 분야 = 스키마 한 덩어리 추가**로 끝나도록 만드는 것이 이 설계의 목적이다. 고객은 필수 최소 + "모름" 안전장치로 부담이 적고, 업체·관리자는 분야별로 구조화된 정확한 정보를 받는다.
