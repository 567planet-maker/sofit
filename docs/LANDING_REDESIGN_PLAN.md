## PART 1. 토스 랜딩페이지 구조 분석

### 토스가 쓰는 공식

토스는 "기능 나열"이 아닌 **"이야기"**로 설계한다.

```
① 강렬한 첫인상 (Hero)
② 사용자의 고통을 언어로 대신해준다 (공감)
③ "우리가 해결했습니다" 기능 스포트라이트 (Feature × 3)
④ 숫자로 신뢰를 증명한다
⑤ 행동을 유도한다 (CTA)
```

### 토스의 디자인 원칙

| 원칙 | 설명 |
|------|------|
| **한 섹션 = 한 메시지** | 섹션에 정보를 욱여넣지 않는다 |
| **비주얼이 카피를 이긴다** | 텍스트 설명 대신 실제 화면이 좌/우에 항상 등장 |
| **다크·라이트 교차** | 배경색이 바뀔 때 사용자가 "페이지가 넘어가는" 느낌 |
| **극도로 큰 타이포그래피** | H1은 최소 56px, 감정을 실은 짧은 문장 |
| **왼쪽 정렬 텍스트** | 중앙 정렬이 아닌 좌측 정렬로 신문처럼 읽히게 |
| **단일 CTA** | 섹션당 버튼 하나. 선택지를 주지 않는다 |
| **실제 UI 목업** | 앱 화면 스크린샷이 주인공. 일러스트·이모지 쓰지 않음 |

### 토스의 스크롤 내러티브

```
사용자가 페이지를 내리면서 느끼는 감정 흐름:

"맞아, 나 이거 불편했어"   →   "오, 이렇게 해결해줘?"
→   "이것도?"   →   "이것까지?"   →   "믿을 수 있겠다"
→   "지금 당장 써봐야겠다"
```

## PART 2. 소핏 타깃 분석

### 주 타깃: B2B 발주 담당자
- 카페·식당·기관의 인테리어 담당자 or 원장
- 쇼파 교체 / 빌트인 가구 제작 발주 경험 있는 사람
- 공장 찾는 게 너무 번거로워서 아는 업체 그냥 쓰거나 포기한 경험

### 부 타깃: 공장 파트너
- 전국 가구·소파 제조 공장
- 안정적인 신규 발주처를 원함

### 소핏의 핵심 가치
> "공장 수소문 → 개별 견적 → 납기 확인"을 한 번에 해결

## PART 5. 섹션별 상세 설계

### S1. Hero — "첫인상"

**배경**: 디자인 이미지 확정 (연한 블루-그레이, 3D 오브젝트 산개)

**헤드라인** (H1, 56px+, 굵게)
```
인테리어의 모든 공정
Sofit에서 쉽고 투명하게
```
- "Sofit" 부분만 소핏 블루 (#0064FF) 강조

**서브타이틀** (18px, 회색)
```
기획부터 완공까지, 검증된 협력업체를 한 곳에서 연결합니다.
```

**CTA 버튼** (App Store / Google Play → 교체)
```
[ 견적 무료로 요청하기 →  ]     [ 공장 포트폴리오 보기 ]
   다크 필 버튼 (블랙)             테두리만 버튼 (투명)
```

**레이아웃**: 텍스트+버튼은 화면 중앙 or 살짝 하단 중앙. 3D 오브젝트들이 주변을 감싸는 구조.

### S2. 공감 섹션 — "발주, 이렇게 힘드셨죠?"

**배경**: #0A0A0A (다크), 텍스트만, 이미지 없음

**구조**: 스크롤 내릴수록 고통점이 하나씩 순차적으로 나타남

```
[작게, 회색]  인테리어 발주, 이런 경험 있으신가요?

[크게, 흰색, 64px]  공장 연락처를 수소문하느라
                    몇 주가 지나갔습니다.

[크게, 흰색, 64px]  견적서마다 형식이 달라
                    비교 자체가 불가능했습니다.

[크게, 흰색, 64px]  발주 후 진행 상황을
                    확인할 방법이 없었습니다.

[소핏 블루, 48px]   이제 소핏이 해결합니다.
```

**애니메이션**: 스크롤 진입 시 각 문장이 아래에서 위로 페이드인 (0.3초 딜레이 차등)

**토스 참고**: 토스의 "금융이 너무 복잡했잖아요" 섹션과 동일한 방식

## PART 8. 스크롤 애니메이션 설계

### 기본 진입 효과 (모든 섹션 공통)
```
진입 전: opacity 0, translateY(32px)
진입 후: opacity 1, translateY(0)
전환시간: 0.6s ease-out
```

### 공감 섹션 (S2) — 순차 등장
```
고통점 1 → 0.0s 딜레이
고통점 2 → 0.25s 딜레이
고통점 3 → 0.50s 딜레이
해결 문장 → 0.80s 딜레이
```

### Feature 섹션 (S3~S5) — 텍스트/목업 분리 진입
```
텍스트: 왼쪽에서 슬라이드 인 (translateX -40px → 0)
UI 목업: 오른쪽에서 슬라이드 인 (translateX 40px → 0)
0.15s 딜레이 차이
```

### 신뢰 숫자 (S6) — 카운트업
```
뷰포트 진입 → requestAnimationFrame으로 1.5초 카운트업
50+ : 0 → 50
300+ : 0 → 300  
24: 0 → 24
```

### 구현 방식
```tsx
// src/hooks/useInView.ts — Intersection Observer 커스텀 훅
// threshold: 0.15 (요소 15% 진입 시 트리거)
// 외부 라이브러리 없이 Tailwind transition 클래스 전환
```
