# 견적 요청 양식 현황 (현재 = 쇼파 전용)

> 목적: 현재 견적 요청 양식의 구조·필드·관련 코드/DB를 정리한 문서.
> 이 문서를 기반으로 "쇼파 전용 → 인테리어 전체 섹터"로 견적 범위를 확장한다.
> 작성 기준일: 2026-06-27

---

## 1. 한눈에 보기

- 현재 견적 양식은 **5단계 스텝(Wizard)** 형태이며, **Step 2~4가 사실상 "쇼파"에 종속**되어 있다.
- Step 1(현장 정보)·Step 5(일정)는 품목과 무관한 범용 필드라 재사용 가능.
- 데이터는 단일 테이블 `quote_requests`에 **플랫(flat)하게** 저장된다. (품목별 분리 구조 없음)
- 쇼파 전용 필드: `sofa_type`, `sofa_count`, `seat_count`, `backrest_height`, `has_armrest`, `cushion_type`, `frame_structure`, `flame_retardant`, `waterproof`, 그리고 규격/자재 일부.

### 확장 시 핵심 과제
1. **품목(섹터) 선택 단계 추가** — 쇼파 / 가구·붙박이 / 바닥재 / 벽·도장 / 조명 / 주방 / 욕실 등.
2. **품목별 동적 필드** — 선택한 섹터에 따라 Step 2~4의 입력 항목이 달라져야 함.
3. **DB 구조 결정** — (A) `quote_requests`에 컬럼 계속 추가 vs (B) `category` + JSONB 상세 컬럼 vs (C) 품목별 자식 테이블(`request_items`).
4. **표시(View) 3곳** 동기화 — 고객/공장/관리자 상세 페이지가 모두 쇼파 필드를 하드코딩 중.

## 5. 데이터 흐름

```
[고객] QuoteRequestForm (localStorage 자동저장: key "sofit_quote_draft")
   │  submitQuoteRequest(data, files)
   ▼
[서버 액션] quote-request.ts
   │  1. 인증/고객 확인
   │  2. quote_requests INSERT (status='submitted', ...data 전체 스프레드)
   │  3. request_files INSERT (업로드 파일 메타)
   │  4. chat_rooms INSERT (type='customer_sofit' 자동 생성)
   │  5. admin 전원에게 notifications INSERT
   ▼
redirect → /customer/request/submitted?id=...
```

> ⚠️ INSERT가 `...data`로 **객체 전체를 그대로 펼쳐 넣음** ([quote-request.ts:83-90](../src/app/actions/quote-request.ts#L83-L90)).
> 즉 `QuoteRequestDraft`의 키 = `quote_requests` 컬럼명이 1:1 일치해야 함. 확장 시 이 계약을 유지하거나 변환 로직을 추가해야 한다.

## 6. DB 구조 (현재) — 단일 플랫 테이블

`quote_requests` 한 테이블에 현장·제품(쇼파)·규격·자재·일정 필드가 **모두 컬럼으로** 존재.
품목 구분 컬럼(`category` 등)이 **없음** → 현재는 "모든 요청 = 쇼파"로 암묵적 가정.

```
quote_requests
├─ 현장(필수): company_name, contact, site_name, address, site_manager, available_time
├─ 현장(선택): business_type, has_parking, floor, has_elevator
├─ 제품(쇼파): sofa_type, sofa_count, seat_count, backrest_height,
│              has_armrest, cushion_type, frame_structure, flame_retardant, waterproof
├─ 규격:      total_length, total_width, total_height, seat_height,
│              seat_depth, wall_length, corner_angle
├─ 자재:      fabric_type, inner_material, frame_material, color_code
├─ 일정:      needs_measurement, install_hours, measurement_date,
│              production_start, production_end, delivery_date, install_date, as_available_date
└─ 상태:      status, admin_note, created_at, updated_at
```

연관 테이블: `request_files`(첨부), `matches`(공장 매칭), `factory_quotes`(공장 견적서), `chat_rooms`, `status_logs`.

## 7. 확장(인테리어 전체 섹터) 시 검토 포인트 — Claude AI 협의용

### A. 섹터(품목) 후보
> 어떤 인테리어 섹터까지 받을지 결정 필요. 예시:
> 쇼파/가구, 붙박이장·수납, 바닥재(마루/타일/장판), 벽체·도장·도배, 천장·몰딩,
> 조명·전기, 주방(싱크/가구), 욕실(위생도기/방수), 창호·중문, 필름·시트, 철거·설비.

### B. UI 구조 선택지
- **(1) 섹터 선택 0단계 추가** → 선택한 섹터에 맞는 Step 2~4 동적 렌더링.
- **(2) 다중 섹터 선택** → 한 요청에 여러 품목(예: 바닥+도장+조명) 포함 가능하게.
- 공통 Step(1 현장, 5 일정)은 그대로 재사용.

### C. 데이터 모델 선택지
- **(A) 컬럼 추가**: 가장 단순하나 섹터 늘수록 컬럼 폭증 + NULL 다수. 비추천.
- **(B) `category` + `details JSONB`**: 공통 필드는 컬럼, 섹터별 상세는 JSONB. 유연·빠름. **유력**.
- **(C) `request_items` 자식 테이블**: 한 요청에 여러 품목 라인. 다중 섹터/정규화에 유리, 쿼리·표시 복잡.

### D. 영향받는 화면 (반드시 동기화)
- 폼 1곳: `QuoteRequestForm.tsx`
- 상세 표시 3곳: 고객/공장/관리자 (모두 쇼파 필드 하드코딩 → 섹터별 렌더링으로 교체)
- 서버 액션 1곳: `submitQuoteRequest` (검증/INSERT 계약)
- 타입 2곳: `QuoteRequestDraft`, `QuoteRequest`
- DB 마이그레이션 신규 추가

### E. 부가 정리거리
- 규격 단위 mm/cm 불일치 정리.
- `metadata.title`/`description` 등 "쇼파·빌트인" 카피 문구 업데이트.
- `factory_portfolios.category`(`sofa|builtin|other`)와 섹터 분류 정합성.
