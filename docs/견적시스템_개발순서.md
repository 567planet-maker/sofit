## ⚠️ 설계안 대비 확정·보정 사항 (착수 전 필독)

설계안을 실제 코드에 맞추며 결정/수정한 항목. **설계안 원문보다 이 문서가 우선한다.**

| # | 설계안 원안 | 확정/보정 | 이유 |
|---|-------------|-----------|------|
| 1 | status를 `draft/submitted/matched/closed/cancelled` 5단계로 **교체** | **기존 8단계 유지 + 맨 앞에 `draft`만 추가** | 기존 8단계(`submitted→reviewing→matching→quote_arrived→negotiating→contracted→in_progress→completed`)가 `status.ts`·StatusBadge·관리자 StatusChanger·고객 상세 분기(`status==='quote_arrived'` 등)·`status_logs`에 연결됨. 교체 시 견적도착/계약/시공 UX가 깨짐. → CHECK 제약에 `draft`만 추가. |
| 2 | (언급 없음) | **채팅방 자동생성을 새 `submitQuoteRequest`로 이동** | 현재 [quote-request.ts:110](../src/app/actions/quote-request.ts#L110)이 제출 시 `customer_sofit` 방 생성. draft-first에선 draft가 아니라 **제출(status 전환) 시점**에 만들어야 함. |
| 3 | `address text → jsonb` | **타입 전환 영향 명시** | [customer/requests/[id]/page.tsx:202](../src/app/customer/requests/[id]/page.tsx#L202) 등에서 `req.address`를 문자열로 렌더 중 → jsonb 전환 시 깨짐. Phase 5/6에서 함께 처리. |
| 4 | 공장↔분야 매칭(Phase 7 전제) | **공장 분야 태그 모델을 선행 블로커로 격상** | 현재 분야 데이터는 [factory.ts:464](../src/app/actions/factory.ts#L464)의 `'sofa'|'builtin'|'other'` 3종뿐. 27개 분야 매칭의 핵심 가치가 여기 걸림. |
| 5 | 라이브 시스템 인플레이스 개조 | **새 폼은 피처 플래그/별도 라우트 뒤에서 개발 → Phase 6 이관 완료 시 스위치** | Phase 4(새 폼)가 Phase 6(이관)보다 먼저 나가면 신규 제출은 items에, 구 폼/컬럼이 공존 → 이중쓰기 혼란. |
