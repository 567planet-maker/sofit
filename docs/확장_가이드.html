1. 현재 상태 진단 — 무엇이 하드코딩 되어있는가

핵심 요약

현재 코드는 두 가지 레이어에서 분야(카테고리)가 고정되어 있습니다.
① 포트폴리오 카테고리 — 쇼파·빌트인·기타 (3개 고정)
② 견적 요청 폼 필드 — 쇼파 전용 컬럼 14개가 DB에 고정
이 두 레이어의 수정 난이도가 크게 다릅니다.

1-1. 포트폴리오 카테고리 (쉬움)

공장이 포트폴리오를 등록할 때 태깅하는 분야 구분입니다. 현재 3개소에 하드코딩되어 있습니다.

src/app/portfolios/page.tsx line 12–16 CATEGORY_LABELS 객체

src/app/actions/factory.ts line 291 'sofa' | 'builtin' | 'other' 리터럴
DB: factory_portfolios.category CHECK 제약 CHECK (category IN ('sofa', 'builtin', 'other'))

1-2. 견적 요청 폼 (어려움)

현재 quote_requests 테이블에는 쇼파 전용 컬럼이 14개 고정으로 존재합니다. 분야를 추가할 때마다 이 구조를 바꿔야 합니다.

컬럼 의미 다른 분야에서는?
sofa_type 쇼파 형태 도배라면 "도배 종류", 페인트라면 "페인트 종류"
sofa_count 수량 분야마다 다른 의미
cushion_type 쿠션 타입 타 분야 불필요
flame_retardant 방염 여부 일부 분야 공통
fabric_type 원단 종류 도배: "벽지 종류", 페인트: 불필요
total_length/width/height 규격 (mm) 철거·에어컨은 면적(㎡), 전선은 회로수
⚠️ 핵심 문제: 분야마다 필요한 정보가 완전히 다릅니다. "도배" 견적에는 벽면 면적이 필요하고, "에어컨 설치"에는 실내기 대수·BTU가 필요합니다. 쇼파용 컬럼 구조로는 이를 수용할 수 없습니다.

4. Phase 2 — 견적 요청 폼 다중 분야 지원 (핵심 리팩토링)

🔴 주의: 이 작업은 DB 마이그레이션을 포함하며, 기존 데이터를 보존하면서 진행해야 합니다. 반드시 Supabase 백업 후 작업하세요.

4-1. 현재 구조의 문제

현재 (나쁜 구조) quote_requests 테이블 ├── company_name (현장, 공통) ├── address (현장, 공통) ├── sofa_type ← 쇼파 전용 ├── sofa_count ← 쇼파 전용 ├── cushion_type ← 쇼파 전용 ├── frame_structure ← 쇼파 전용 ├── fabric_type ← 쇼파 전용 └── ... (14개 쇼파 전용 컬럼) 분야 추가 시 문제: → 도배를 추가하면: wall_area, pattern_type 컬럼 추가 → 에어컨을 추가하면: ac_units, btu_required 컬럼 추가 → 철거를 추가하면: demolition_area, waste_type 컬럼 추가 → 결국 quote_requests 테이블에 컬럼이 50개 이상으로 폭증 → 대부분의 행에서 대부분의 컬럼이 NULL

4-2. 권장 해결 방법: JSONB 방식

개선된 구조 quote_requests 테이블 ├── company_name (공통) ├── address (공통) ├── service_category ← "sofa" | "wallpaper" | "hvac" | ... ├── spec_data JSONB ← 분야별 상세 정보를 JSON으로 저장 └── (기존 공통 컬럼 유지: floor, has_parking, has_elevator 등) spec_data 예시 — 쇼파: { "sofa_type": "L형", "sofa_count": 3, "cushion_type": "스프링", "fabric_type": "가죽", "total_length": 3200 } spec_data 예시 — 도배: { "wall_area": 85.5, "wallpaper_type": "실크", "pattern": "무지", "include_ceiling": true } spec_data 예시 — 에어컨: { "ac_count": 4, "indoor_type": "4방향 카세트", "btu_per_unit": 12000, "pipe_length": 15 }

4-3. DB 마이그레이션 SQL

-- 파일: supabase/migrations/003_multi_category.sql

-- 1. service_category 컬럼 추가
ALTER TABLE quote_requests
  ADD COLUMN service_category text DEFAULT 'sofa'
    CHECK (service_category IN (
      'sofa', 'builtin', 'demolition', 'electrical',
      'fire_safety', 'ceiling', 'hvac',
      'wallpaper', 'paint', 'flooring', 'lighting', 'other'
    ));

-- 2. spec_data JSONB 컬럼 추가 (분야별 상세 정보)
ALTER TABLE quote_requests
  ADD COLUMN spec_data jsonb;

-- 3. 기존 쇼파 데이터 마이그레이션 (기존 컬럼 → JSON)
UPDATE quote_requests
SET
  service_category = 'sofa',
  spec_data = jsonb_build_object(
    'sofa_type',       sofa_type,
    'sofa_count',      sofa_count,
    'cushion_type',    cushion_type,
    'fabric_type',     fabric_type,
    'total_length',    total_length,
    'total_width',     total_width,
    'total_height',    total_height
    -- ... 나머지 쇼파 컬럼
  )
WHERE sofa_type IS NOT NULL OR sofa_count IS NOT NULL;

-- 4. (선택적) 기존 쇼파 전용 컬럼 제거 — 충분히 안정화된 후에 실행
-- ALTER TABLE quote_requests DROP COLUMN sofa_type, DROP COLUMN sofa_count, ...;
📌 전략: 기존 컬럼을 즉시 삭제하지 말고, spec_data로 이전 후 충분히 검증된 다음에 삭제합니다. 이렇게 하면 롤백이 가능합니다.

ALTER TABLE factories

  PRIMARY KEY (factory_id, category)
